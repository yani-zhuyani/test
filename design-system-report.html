01 · Problem

为什么 AI 会输出不一致的 B 端界面？

快手 B 端有大量专有设计约定，没有规范文档，AI 只能凭通用前端知识"猜"，导致每次生成结果都不一样。

❌ 规范不可及
AI 无法凭直觉得知内部专有规定

"弹框宽度 400px"、"标签末尾必须加冒号"、"胶囊标签页禁止用于上级导航"——这些是快手内部规定， 通用前端知识里根本不存在，不查文档就必定写错。

Without Skill：快手专有规范命中率 0/15 = 0%
❌ 每次都不一样
没有约束就靠"发挥"，一致性为零

弹框宽度一次 500px 一次 600px；按钮顺序时左时右；分页有时有跳转框有时没有。 AI 的创造力在规范场景里是负资产。

标准差无法量化：每次生成结果不可预期
⚠️ 只有正例，没有约束
AI 会对正例举一反三出奇怪变体

"应该用主色蓝" → AI 会发挥出渐变蓝、半透明蓝、深蓝。 反例（禁止项）和决策树才是让 AI 不乱发挥的真正约束。

⚠️ 缺决策树
遇到模糊场景，AI 只能随机选择

弹框 vs 抽屉 vs 页内展开？主操作按钮放哪里？这类判断需要明确的决策规则， 没有就靠运气。

🎯
根本原因

约 71% 的快手 B 端规范是内部专有约定， 与通用前端知识完全无重叠。没有 Skill，这些规范对 AI 来说就是"不存在"。 解法不是让 AI 更聪明，而是把规范知识结构化地喂进去。

03 · Skill Framework

规范 Skill 的 4 层内容框架

让 AI 稳定输出的核心是"约束层"，而不只是"内容层"。4层结构从设计语言到可验证 Checklist 逐层增强约束力。

LAYER 0
设计语言
全局 Token
精确到元素 × 状态的 Token 绑定

不是"蓝色"，而是"主要按钮正常态背景 = #2563F4"

✓ 色彩 Token（含状态变化）
✓ 间距系统（4px 基准网格）
✓ 字体体系（字号+行高+粗细）
✓ 圆角/阴影（按组件类型）
LAYER 1
组件规范
精确 + 反例
正例 + 反例库 + 组合约束三合一

AI 会对正例举一反三出变体，但对"❌不允许"几乎不会违反。反例库是关键缺失项。

✓ 尺寸表（各变体精确 px）
✓ 状态矩阵（Normal/Hover/Active/Disabled）
⭐ 反例库（❌禁止用法集中管理）
✓ 跨组件组合约束（按钮组顺序）
LAYER 2
页面模式
模板先行
直接给框架，比让 AI 推导规则更稳

3-5 个典型页面模板，含完整区域划分 + 精确间距标注。模板 > 规则。

✓ 天工数据平台（已有）
✓ APP 分析看板（已有）
✓ AB Test 管理页（已有）
+ 通用列表页/设置页（待补充）
LAYER 3
决策树
消除歧义
遇到模糊场景，AI 需要判断逻辑而不是猜

弹框 vs 抽屉 vs 页内展开？主操作按钮放哪里？没有规则就靠随机，有了决策树就能精确选择。

⭐ 弹框 vs 抽屉 vs 页内（判断条件）
✓ 主操作按钮位置规则
✓ 字段类型选择（输入框 vs 选择器）
LAYER 4
Checklist
可验证
P0/P1/P2 分级核查，判断条件是 CSS 属性值

按页面类型分类。P0=必须通过，P1=强烈建议，P2=可接受偏差。判断是客观属性值，不依赖主观判断。

⭐ Eval 驱动持续验证

核心洞察

Without Skill 通过的所有断言，100% 来自通用前端知识（按钮高亮、省略号分页等），与快手内部规范完全无关。 快手专有规范（标签冒号、弹框400px、胶囊禁用规则……）Without Skill 命中率精确为 0%。 这证明：Skill 对内部规范场景是不可替代的基础设施，不是锦上添花。

🎯
最终目标

将快手内部 B 端设计规范的每一条约定转化为 AI 可精确执行的知识结构。 不再依赖设计师逐页走查，不再依赖开发者记忆规范细节。 任何人让 AI 生成一个 B 端页面，输出结果就应该和设计师亲手画的稿子一样。
