4 条关键洞察

Insights
  • 真正断裂的不是单页，而是 一条任务链路。
  • 进入系统后，用户先卡在 找入口，再卡在 发布总览缺失。
  • 从“步骤详情”到“实例详情”再到“监控”是情绪持续下滑的核心链路。
  • 止损阶段能回升，不是因为系统足够好，而是因为老用户已经学会了绕路。

事情 01 · 入口与链路割裂是高频共性反馈

事情（事实）

3 位受访用户都直接提到入口分散或跨系统跳转问题，涉及 KDev、天琴、事件中心、天问、CMDB 等多个触点。

证据来源

钱立勋、郑佳俊、肖干 3 份访谈原话

判断性质

用户反馈已重复出现，可视为已确认问题

置信度

高

事情 02 · 发布执行阶段是当前最低情绪区间

事情（事实）

从“步骤详情”到“实例详情”再到“监控 / 日志”的连续链路，承接了最多的负面描述，包括看不全、切换重、难联查、难判断异常。

证据来源

钱立勋、肖干关于发布总览、实例查看、监控跳转的集中反馈

判断性质

负面反馈集中落在同一连续任务链路

置信度

高

推测 01 · 新手用户会更早在“定位对象 / 添加环境”阶段掉队

推测

当前受访者多为已有经验的研发同学，仍然反复提到入口分散、概念门槛和路径难找；因此可推测对低经验用户而言，低谷可能比当前地图更早出现。

依据

成熟用户已经会绕路，仍持续反馈“找入口难”“添加环境重”

影响判断

说明当前地图对新手痛点可能还是偏保守

置信度

中

推测 02 · 如果先做“发布总览 + 联查闭环”，整体体验提升会最明显

推测

最低情绪点集中在发布执行与排障交界处，而这里同时承接了最多 P0 问题与最高频任务，因此优先做发布总览与联查闭环，预计会带来最直接的体验改善。

依据

情绪最低点、问题清单、机会点三处信号高度重合

对应动作

优先整合步骤、实例、监控、日志、变更信息

置信度

中

验证清单

Validation

这些结论里，哪些已经足够稳，哪些还要继续补证据，要单独拎出来，避免后续方案建立在未经验证的假设上。

已验证

  • 入口分散、跨系统跳转割裂，已被 3 位受访用户重复提及。
  • 发布阶段缺少总览、实例与步骤割裂，是当前最集中、最明确的体验问题之一。
  • 问题解决高度依赖经验与群协作，自助闭环不足，已在共性问题与原始问题中多次出现。

待验证

  • 新手用户是否会在“定位目录 / 服务”“添加环境”阶段更早出现情绪低谷。
  • “发布总览页”上线后，是否真的能显著减少步骤 / 实例 / 监控切换成本。
  • 查询助手是否适合承接更多排障动作，还是只适合信息查询场景。

建议补充验证

  • 补 2-3 位低经验研发样本，重点观察首次创建服务、首次加环境、首次发版。
  • 做“发布总览页”低保真原型测试，测任务完成时间、切换次数、异常定位速度。
  • 区分查询类能力与高风险执行类能力，验证用户对 AI 辅助边界的接受度。

机会点与建议

Priorities

P0

  • 做统一任务入口：我要发布 / 查变更 / 建环境 / 看实例。
  • 在部署页提供实例汇总、步骤关联、发布进度总览。
  • 打通实例 / 日志 / 监控 / 变更的一站式排障链路。
  • 重做添加环境流程，减少层级与重复输入。

P1

  • 增加我的服务 / 收藏服务 / 最近使用，降低定位成本。
  • 审核人、责任人、权限缺失做成显式推荐与提示。
  • 优化变更事件查询、实例检索性能与空态反馈。
  • 把监控视图改造成“发布决策视图”，而非专业图表堆叠。

P2

  • 为新手提供任务式引导，补足平台概念认知。
  • 统一平台角色边界文案，减少系统命名混淆。
