3 种方案，选一个方向

设计稿 Figma → 前端页面截图 → Diff → 报告。核心差异在于"谁来运行"

🔌
浏览器插件
设计师本地运行

浏览器插件（Chrome Extension）

设计师安装插件，在浏览器里打开前端页面，点一下按钮自动拉取 Figma API 获取对应 Frame，截图对比，悬浮显示 diff 高亮 + 问题列表

优势

  • 零部署，设计师自助
  • 实时叠加 overlay 直观
  • 无需后端

劣势

  • 需要发布到应用商店或内部分发
  • 跨域/截图有限制
⚡
CLI + CI 工具
集成到流水线

CLI 命令行工具 + CI 集成

一条命令 design-review --url http://staging.xxx.com --figma-url xxx，Playwright 截图 → Figma API 拉图 → pixelmatch 对比 → 生成 HTML 报告 + 发 PR comment + 推 IM

优势

  • CI 自动触发，PR 节点天然契合
  • 技术最成熟，可精准控制
  • 报告可存档

劣势

  • 需要接入 CI/CD
  • 设计师不能自助触发
🌐
Web 平台
团队共享，最完整

内部 Web 平台

搭建一个 Web 服务，设计师填入 Figma URL + 页面 URL，后端用 Playwright 截图对比，结果展示在平台上，支持历史记录、多人协作标注、趋势追踪

优势

  • 功能最完整
  • 设计师自助，团队共享
  • 可沉淀历史数据

劣势

  • 需要部署和维护后端
  • 开发成本最高

💡 我的推荐：方案 B（CLI + CI）— 和你们 PR 节点触发的需求完美匹配，技术方案成熟，先跑通核心链路，后续可以再加 Web UI
