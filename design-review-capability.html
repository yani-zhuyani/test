Layer 1 · 视觉走查
对比设计稿与前端截图，定位颜色/间距/字号等视觉属性偏差，量化还原度（0-100分）

✅ 已可实现
▼
Skill 触发词
走查 [URL] [Figma-URL] 视觉走查 还原度检查 design review 设计比对
执行步骤（5步）
1
解析 Figma URL → 获取 Frame 图像
提取 file-key + node-id，调用 Figma REST API /images 导出 PNG（@2x）
GET /v1/images/{fileKey}?ids={nodeId}&scale=2&format=png
2
Playwright 截取前端页面
Headless Chromium，设定视口尺寸（默认 1440×900），全页截图或指定区域
page.screenshot({ fullPage: true, clip: {...} })
3
图像对齐 + pixelmatch 逐像素对比
jimp resize 对齐尺寸 → pixelmatch（threshold=0.1）→ 生成差异热力图
pixelmatch(img1, img2, diff, w, h, {threshold:0.1})
4
AI Vision 分析差异区域
热力图 + 原图 → Claude Vision / GPT-4V → 识别差异类型，生成文字描述
分类：颜色偏差 / 间距偏差 / 字号偏差 / 布局差异
5
生成报告 + 多渠道推送
HTML 报告 + 还原分（0-100）+ 问题清单，推送飞书卡片 / PR Comment
还原分 = 100 - (diffPixels / totalPixels) × 100
输入参数
figmaUrl 必填
Figma Frame 链接，含 node-id 参数
pageUrl 必填
前端页面地址（支持 localhost / staging）
viewport 可选
默认 1440×900，支持 375×812（移动端）
threshold 可选
像素差异容忍度，默认 0.1（0~1）
输出产物
🔴 差异热力图 PNG
红色高亮差异区域，可定位到问题坐标
📊 还原度分数（0-100）
颜色/间距/字号/布局 分项打分
📋 问题清单（JSON + HTML）
问题类型、坐标、严重程度、AI描述
💬 飞书卡片 + PR Comment
自动推送，附缩略图和问题摘要
⚡
降级策略：Figma API 失败 → 提示手动上传截图；Playwright 失败（需登录）→ 提示前端提供截图；AI Vision 超时 → 跳过归因，只输出像素 Diff 热力图 + 坐标数据。

🚧 Beta
▼
Skill 触发词
交互走查 状态走查 验证交互 interaction review
交互剧本 DSL（YAML）
设计师填写操作序列，描述用户行为与预期状态：
# 示例：登录表单走查 steps: - action: click selector: "#submit-btn" expect: error-state figma_frame: "Login/Error" - action: type selector: "#email" value: "user@test.com" expect: focus-state - action: hover selector: ".btn-primary" expect: hover-state figma_frame: "Button/Hover"
可检测状态类型
🔵 Hover 态
鼠标悬停时的颜色变化、阴影反馈
🟡 Focus 态
输入框获取焦点时的边框、光标样式
🔴 Error 态
表单校验失败时的红色提示、错误文案
✅ Success 态
操作成功后的 Toast、状态图标变化
⏳ Loading 态
异步请求期间的 Skeleton / Spinner
📄 Empty 态
无数据时的空态插图和引导文案
🔗 跳转路径
点击后 URL 是否与设计预期一致
🎬 动效时机
过渡动画时长是否符合设计规范（ms）
输出产物
🎞 状态截图序列
每步操作前后截图对比，左设计 / 右实现
🎬 录屏 GIF
Playwright 录制 → ffmpeg 转 GIF，嵌入报告
❌ 缺失状态清单
哪些状态未实现，对应 Figma Frame 名称
⏱ 动效时长偏差
实际 transition-duration 与设计规范差值
⚠
前置条件：设计师需维护 YAML 剧本文件。可选：AI 自动从 Figma Variables 推断状态树（开发中）。

对确定性视觉问题（颜色/间距/字号）自动定位源码、生成最小 patch、截图验证、提 PR
问题分级 AST 源码定位 最小 patch 3次迭代验证 自动 PR

Skill 触发词
自动修复 走查并修复 design fix auto patch
修复流程（5步）
1
问题分级
L1/L2 问题清单分为「确定性」和「主观判断」，只有确定性问题进入自动修复
2
AST 定位源码
grep → tree-sitter AST → 定位对应 CSS class / Token 引用行
支持：CSS / SCSS / Tailwind / CSS-in-JS
3
生成最小 patch
只修改目标属性值，不改结构。优先修改 Token 引用而非魔法值
#6366f1 → var(--color-primary)
4
截图验证（最多3次迭代）
patch 后重跑 L1 Diff，还原分提升 ≥5 则通过；否则 AI 重新分析调整
通过条件：还原分提升 ≥ 5 分
5
创建 PR + 通知闭环
PR 附走查报告、修复前后对比图、还原分变化。飞书通知设计师确认
问题分级规则
🤖 AI 自动修（确定性）
颜色值偏差（hex / rgb / hsl 可精确对比）
间距偏差（padding / margin / gap 具体 px）
字号偏差（font-size 具体 px / rem 值）
字重偏差（400/500/600/700）
圆角 / 透明度具体值偏差
👤 人工处理（主观/结构性）
布局结构差异（flex/grid 方向改变）
组件缺失 / 替换为错误组件
交互行为缺失（状态未实现）
响应式在特定断点异常
前置条件 & 依赖
Design Token 规范化
项目用 CSS 变量 / Tailwind Config 管理颜色间距，Token 越规范修复准确率越高
代码仓库访问权限
需 GitHub Token，至少 repo 读写权限
staging 环境可访问
支持 Cookie 注入绕过登录鉴权
依赖 L1 输出
L3 以 L1 问题清单（JSON）作为输入，不可单独触发
⚠
关键约束：3次迭代后仍未通过 → 放弃自动修复，移入人工处理清单，附 AI 修复尝试记录。

问题处理决策规则
问题类型 来源层 可自动修 处理路径 示例
颜色值偏差L1✅ 是AST 定位 → patch color 值 → 验证#6366f1 → #4f46e5
间距偏差L1✅ 是定位 padding/margin → patch px 值padding:16px → 12px
字号/字重偏差L1✅ 是定位 font-size/weight → patch14px → 13px
圆角/透明度偏差L1✅ 是定位 border-radius/opacity → patch6px → 4px
Hover/Focus 状态缺失L2⚠ 部分AI 生成 CSS :hover/:focus 规则，需人工确认缺少 :hover 颜色变化
Error/Empty 状态缺失L2❌ 否输出问题单 → 分配给前端开发空态组件未实现
布局结构差异L1❌ 否人工分析 → 前端重构flex→grid 方向错误
组件替换错误L1❌ 否标注组件名 → 前端替换用了旧版 Button 组件
动效时长偏差L2✅ 是定位 transition-duration → patch ms 值300ms → 200ms
跳转路径错误L2❌ 否输出问题单（含预期路由）→ 前端修复/user → /profile
