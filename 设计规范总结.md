## 一、设计系统概述

Google Style Guides 是一套覆盖多种编程语言及文档写作的**统一风格规范体系**，旨在：

- 保持代码库的**一致性（Consistency）**
- 提升代码的**可读性（Readability）**
- 降低大型项目的**维护成本（Maintainability）**
- 促进**跨团队协作**效率

### 覆盖范围

| 类型     | 语言 / 格式                                                    |
| -------- | -------------------------------------------------------------- |
| 系统语言 | C++、C#、Objective-C                                          |
| 脚本语言 | Python、Shell、R                                              |
| Web 语言 | HTML/CSS、JavaScript、TypeScript、AngularJS                   |
| 后端语言 | Java、Go                                                      |
| 数据格式 | JSON、XML                                                     |
| 文档     | Markdown                                                      |
| 其他     | Lisp、Vim Script                                              |

## 二、核心设计原则

### 2.1 激进的简洁性（Radical Simplicity）

> 源自 [docguide/philosophy.md](docguide/philosophy.md)

- **可扩展性和互操作性**优先于功能堆砌。简洁带来规模，无修饰的内容带来互通。
- **减少干扰**，让写作更专注、阅读更高效。
- **新特性不应妨碍最简用例**，对不需要它的用户应完全透明不感知。
- 工具面向**普通工程师**设计——那些忙碌、只想快速回去写代码的人。
- **减少上下文切换**让人更愉快。工程师应能用写代码的同一套工具来阅读和编写文档。

### 2.2 可读的源文本（Readable Source Text）

- **纯文本不只是够用，它更优越**。内容与展示分离，应始终能在不依赖渲染器的情况下阅读核心信息。
- **内容与呈现不应混合**。用户不应被迫接触展示层。
- **可移植性和面向未来的设计**，通过保持源文件尽可能人类可读来实现。
- **静态内容优于动态内容**，但**新鲜内容优于过时内容**。

### 2.3 最小可行文档（Minimum Viable Documentation）

- **少量新鲜准确的文档胜过大量残破的文档**。
- **简短实用优于冗长详尽**。大多数用户只需要作者知识的一小部分，但需要快速获取。
- 文档应像测试一样对待：**必要且值得投入**，因为长期有回报。

### 2.4 持续迭代优于追求完美（Better is better than Best）

- **增量改进优于长期争论**。耐心和对不完美的容忍让项目有机生长。
- **不要占着坑不干**（Don't lick the cookie, pass the plate）。想法很多，选择真正能做的。
- 快速迭代是你的朋友。作者要在短期改进时保持生产力，设定较低的每次提交标准，让更多改进发生。

## 三、通用代码规范

### 3.1 命名规范总则

各语言命名风格汇总：

| 语言       | 类/类型         | 函数/方法       | 变量/字段          | 常量              |
| ---------- | --------------- | --------------- | ------------------ | ----------------- |
| C#         | `PascalCase`    | `PascalCase`    | `camelCase` / `_camelCase` | `PascalCase`  |
| Go         | `MixedCaps`     | `MixedCaps`     | `mixedCaps`        | `MaxLength`       |
| R          | `BigCamelCase`  | `BigCamelCase`  | `lowerCamelCase`   | —                 |
| Shell      | `snake_case`    | `snake_case`    | `UPPER_SNAKE_CASE` | `UPPER_SNAKE_CASE`|

**通用命名原则：**

- 名称应**自描述**，避免冗余的缩写
- 避免在使用处显得**重复啰嗦**
- 结合**上下文**考虑，不重复已清晰的概念
- 优先使用**准确**的词汇而非笼统模糊的词

### 3.2 格式化规范

**缩进：**
- C#：2个空格缩进，禁止使用 Tab
- Shell：2个空格缩进
- Go：使用 `gofmt` 工具自动格式化（无需手动）

**行长度：**
- C#：最大 100 字符
- Markdown：建议 80 字符（链接、表格、标题、代码块除外）
- Go：无固定行长限制，过长时优先考虑重构

**括号与空格：**
- 大括号不换行放置（C# 等语言）
- `if`/`for`/`while` 等关键字后加空格
- 括号内侧不加空格

### 3.3 注释规范

**注释类型层级（从低到高）：**

1. **有意义的命名** — 通过良好命名让代码自解释
2. **行内注释** — 解释"为什么"，而非"是什么"
3. **方法/API 注释** — 说明用法、参数、返回值、异常
4. **类/模块注释** — 简要说明整体用途与用法示例
5. **README.md** — 面向新用户的目录引导
6. **设计文档/PRD** — 记录设计决策和权衡

**注释写作原则：**
- 注释应解释**为什么**，不是**是什么**
- 避免重复代码已表达的内容
- 与代码同步更新（同一次提交）
- TODO 注释格式统一

### 3.4 代码组织原则

- **一个文件一个核心类**（C#）
- 模块/类成员按类型分组（嵌套类 → 静态字段 → 实例字段 → 构造函数 → 方法）
- `using`/`import` 声明置于文件顶部，按字母顺序排列
- 减少不必要的抽象层

## 四、各语言规范

### 4.1 Go 代码规范

> 源自 [go/guide.md](go/guide.md)、[go/decisions.md](go/decisions.md)、[go/best-practices.md](go/best-practices.md)

#### 五大风格原则（重要性排序）

1. **清晰性（Clarity）** — 代码目的和原理对读者清晰
   - 从读者角度出发，而非作者视角
   - 既要清楚"代码在做什么"，也要清楚"为什么这样做"
   - 使用更具描述性的变量名、添加必要注释

2. **简洁性（Simplicity）** — 以最简方式完成目标
   - 易于从头到尾阅读
   - 不假设读者了解前置知识
   - 没有不必要的抽象层
   - **最小机制原则**：优先使用核心语言构件 → 标准库 → 核心库，最后才引入新依赖

3. **精炼性（Concision）** — 高信噪比
   - 避免重复代码
   - 避免多余语法
   - 避免不透明命名
   - 避免不必要的抽象

4. **可维护性（Maintainability）** — 易于未来修改
   - API 结构支持优雅扩展
   - 基于问题结构而非代码结构的抽象
   - 完善的测试套件
   - 可预测的命名

5. **一致性（Consistency）** — 与整体代码库风格一致
   - 当发生风格冲突时，一致性可作为打破平局的依据
   - 包内一致性通常最重要

#### 核心规范

- 所有 Go 源文件必须用 `gofmt` 工具格式化
- 多词名称使用 `MixedCaps` 驼峰，不使用下划线（含常量）
- 行长无固定限制，过长优先重构而非折行
- 避免在不必要时使用复杂机制（接口、反射等）

### 4.2 C# 代码规范

> 源自 [csharp-style.md](csharp-style.md)

#### 命名规则

| 元素               | 规则                    | 示例                    |
| ------------------ | ----------------------- | ----------------------- |
| 类、方法、枚举     | `PascalCase`            | `MyClass`、`DoSomething`|
| 本地变量、参数     | `camelCase`             | `localVar`、`paramName` |
| 私有/内部字段      | `_camelCase`（下划线前缀）| `_privateField`       |
| 接口               | `I` + `PascalCase`      | `IMyInterface`          |
| 文件名/目录名      | `PascalCase`            | `MyFile.cs`             |

#### 代码规范

- `const` 和 `readonly` 优先于普通字段
- 禁止使用魔法数字，使用命名常量
- 几乎总是使用 `class` 而非 `struct`，除非类型行为像值类型
- 扩展方法谨慎使用，仅用于核心通用功能
- Lambda 非简单时应提取为命名方法
- 字段初始化器推荐使用

#### 类成员顺序

1. 嵌套类、枚举、委托和事件
2. 静态字段、常量和只读字段
3. 实例字段和属性
4. 构造函数和终结器
5. 方法

*每组内按 Public → Internal → Protected Internal → Protected → Private 顺序排列*

### 4.3 Shell 脚本规范

> 源自 [shellguide.md](shellguide.md)

#### 使用时机

- 适合：简单工具脚本、调用其他工具为主
- 不适合：性能敏感场景、超过100行的复杂脚本
- 超过100行或控制流复杂时，应重写为结构化语言

#### 核心规范

- **仅使用 Bash**，不追求 POSIX 兼容性
- 可执行文件以 `#!/bin/bash` 开头
- 文件扩展名：可执行文件用 `.sh` 或无扩展名，库文件用 `.sh`

#### 命名规范

| 元素        | 规范                |
| ----------- | ------------------- |
| 函数名      | `snake_case`（小写）|
| 普通变量    | `snake_case`（小写）|
| 常量/环境变量 | `UPPER_SNAKE_CASE`|
| 源文件名    | `snake_case`（小写）|

### 4.4 R 语言规范

> 源自 [Rguide.md](Rguide.md)

- 函数名使用 `BigCamelCase`（大驼峰）区别于其他对象
- 私有函数以 `.` 开头（如 `.DoNothingPrivately`）
- 不使用右向赋值（`-> results`）
- 明确使用 `return()` 表达返回意图
- 外部函数使用命名空间限定（如 `purrr::map()`）
- 所有包需要包级别文档文件（`packagename-package.R`）

## 五、文档规范

### 5.1 Markdown 文档规范

> 源自 [docguide/style.md](docguide/style.md)

#### 平衡三大目标

1. **源文本可读且可移植**
2. **文档库可长期跨团队维护**
3. **语法简洁易记**

#### 文档结构规范

推荐文档布局：

```markdown
# 文档标题

简短介绍（1-3句话）。

[TOC]

## 主题一

内容。

## 参见

* https://更多信息链接
```

- 文档以一级标题（H1）开头，之后使用 H2 及以下
- TOC 放在简介之后、第一个 H2 之前
- 使用 ATX 风格标题（`#` 前缀），不使用下划线风格

#### 格式规范

| 规范项目   | 要求                                           |
| ---------- | ---------------------------------------------- |
| 行字符限制 | 80字符（链接、表格、标题、代码块除外）         |
| 标题风格   | ATX 风格（`# 标题`），标题前后各空一行         |
| 标题唯一性 | 每个标题使用唯一且完整的描述性名称             |
| 缩进       | 有序/无序列表使用4空格缩进                     |
| 代码块     | 使用围栏式代码块，并声明语言类型               |
| 链接       | 长链接使用引用式链接，使用有意义的链接文字     |
| 尾部空格   | 不使用尾部空格，需换行时用 `\` 结尾            |
| 优先级     | 强烈优先使用 Markdown 而非 HTML                |

#### 大小写规范

- 保留产品、工具、二进制文件的**原始名称大小写**
- 遵循 [Google Developer Documentation Style Guide](https://developers.google.com/style/) 的标题大小写规范

### 5.2 README 文档规范

> 源自 [docguide/READMEs.md](docguide/READMEs.md)

**必要条件：**
- 文件名必须为 `README.md`（大小写敏感）
- 位于对应代码库的顶级目录中，**不**放在 docs 文档目录内

**必要包含内容：**

1. 本目录/库的用途说明
2. 联系人信息
3. 状态说明（是否废弃、是否对外发布等）
4. 使用方法（示例代码、常用命令等）
5. 相关文档链接

### 5.3 文档最佳实践

> 源自 [docguide/best_practices.md](docguide/best_practices.md)

#### 最小可行文档

- **少量新鲜准确的文档 > 大量残破失效的文档**
- 文档像盆景：活着但频繁修剪
- 写简短有用的文档，删除过时、错误、冗余的内容

#### 与代码同步更新

- 在**同一次提交**中修改文档与代码
- 代码审查时要求同步更新 docstring、头文件、README

#### 清理死亡文档

- 删除已过时的文档
- 方法：缓慢推进、先删肯定错误的、让全组参与决策（保留或删除）

#### 文档层次体系

```
命名（自描述）
  ↓
行内注释（解释为什么）
  ↓
方法/API 注释（说明合约）
  ↓
类/模块注释（概述用途）
  ↓
README.md（目录导航）
  ↓
docs 目录（操作指南）
  ↓
设计文档（架构决策记录）
```

#### 避免重复

- 不要为常见的 Google 技术或流程重写指南，**链接到现有资源**
- 如果指南不存在或严重过时，提交更新或创建 README

## 六、交互与工程规范

### 6.1 代码审查规范

**作为审查者：**
- 合理时立即 LGTM，相信作者会修复注释中的问题
- 提供具体替代方案而非模糊评论
- 重大变更时自己提独立后续 CL，避免 "你还应该..." 式评论
- 仅在 CL 实际让文档变差时才阻止提交

**作为作者：**
- 避免在琐碎争论上浪费精力，早点妥协继续前进
- 必要时引用"更好胜于最好"原则（Better/Best Rule）

### 6.2 版本控制与提交规范

- 代码与文档变更**同一 CL（变更列表）** 提交
- 提交信息应清晰表达变更意图
- 参考：[How to Write a Git Commit Message](https://cbea.ms/git-commit/) 原则适用于所有版本控制系统

### 6.3 规范演进策略

- 规范应服务于**内部一致性需求**，而非追求绝对正确
- 一次性修复所有风格偏差是不实际的，**就近改进**即可
- 新添加的规范条目**必须经过充分讨论**，门槛较高
- 不以新旧文档不一致为由引入额外 churn（不必要的重构）

### 6.4 工具与自动化

- **Go**：强制使用 `gofmt` 进行代码格式化（通过预提交检查强制执行）
- **Shell**：推荐使用 `ShellCheck` 进行静态分析
- **代码搜索兼容性**：格式规范需与代码搜索工具兼容（如80字符行长规范）
- **IDE 集成**：格式规范应与常用 IDE 的格式化行为对齐
