为什么要做这个 Skill 把隐性经验变成显性工作流

用户旅程地图的制作原本非常依赖个人经验、表达能力和临场追问能力。做这个 skill 的意义，是把这套方法从“会做的人脑子里”抽出来，沉淀成别人也能直接调用的标准流程。

不做 Skill 的传统模式 高依赖人工，质量与效率都不稳定

传统做法通常是人工追问、手工整理、临时拼装输出。结果就是收集过程冗长、分析结构不统一、事实与建议容易混写，而且很难分享给别的用户复用。

这个 Skill 的概述 一条从输入到汇报的轻量生产线

它把信息收集、7 条泳道分析、事实/推测/建议标注，以及 Mermaid / Markdown / HTML 报告输出串成一条标准化工作流，确保旅程地图既能快速生成，也能正式汇报。

它带来的价值 更可复用，也更提效

它把多轮沟通压缩成一轮输入，把临时分析变成稳定结构，也让同一套方法既能服务你自己，也能服务其他用户，显著提高复用率与交付效率。

执行摘要

Executive Summary
这个 skill 的本质

它不是只负责“生成一张图”，而是把输入收集、结构分析、证据标注、正式输出串成一条轻量工作流。

解决的核心问题

把原来容易拖长的逐题引导，收敛成一屏输入模板，更适合对外分享和复用。

为什么要重做页面

汇报页不应该继续沿用旅程地图本身的表达结构，而应该更像一份策略说明与能力总览。

下一阶段重点

继续推进模板产品化、模式分层，以及多文件联动维护的收敛。

3 条关键判断

Key Calls
  • 对外使用时，一次性收集模板 比 7 个问题逐条追问更符合真实场景。
  • 正式汇报页应采用能力总览 + 结构图 + 决策项，而不是继续套旅程地图版式。
  • 在当前项目里，最稳妥的方式是直接复用 Figma Board 模板语言，而不是重新发明一套视觉系统。

标准 SOP

Workflow

这条流程带只回答“这个 skill 怎么工作”，而不展开具体业务旅程内容。

01

识别输入模式

判断用户是只有需求、已有材料，还是已有页面/链接。

02

一次性收集信息

优先发统一模板，不再默认进入长链路追问。

03

压成 7 条泳道

把输入转成统一分析结构，便于稳定输出。

04

证据分层

所有结论区分为【事实 / 推测 / 建议】，并带证据来源或置信度。

05

选择输出载体

根据场景输出 Mermaid、Markdown 或 HTML 报告。

使用手册

Usage
场景 1

用户只有一句话需求

直接发标准输入模板，让对方一次性补齐关键信息。

场景 2

用户已有访谈 / 调研材料

优先接收原始材料，先抽事实，再补推测和建议。

场景 3

用户要正式汇报

优先走 HTML 报告模板，保留证据分层、问题清单和结构化可视化。
